# 프론트엔드

요약: 처음 시도하는 프로그래밍 언어과 그로 인해 발생하는 흔한 문제, 소통과 개발 속도의 문제

프론트 담당을 맡으면서, 웹 페이지를 어떤 언어로 구현할지 고민이 있었다. 웹 개발에서 사용할 줄 아는 언어가 HTML/CSS/JS만 있었기 때문이었다.  
수업에서 배운 베이직 웹으로도 페이지를 만들 수는 있었으나, React를 선택하면 완성도가 높고, 실제 실무에서 선택하는 언어이고, 앞으로의 프로젝트에서도 자주 사용하게될 언어임을 고려하여, React와 Typescript, Vite를 기술 스택으로 선택했다. 따라서 아래 애로사항은 처음 배워서 사용하는 과정에서 벌어진 문제가 많다.

### **1\. 환경 설정 및 라이브러리 버전 충돌 (Tailwind CSS)**

* **애로사항:**  
  * npx tailwindcss init \-p 명령어를 입력했을 때 could not determine executable to run 에러가 발생하며 설정 파일이 생성되지 않음.  
  * 원인을 분석해보니 최신 버전인 **Tailwind CSS v4**가 설치되었으나, 사용하려는 명령어와 설정 방식은 **v3** 기준이어서 호환되지 않음.  
* **해결 방안:**  
  * 기존에 설치된 v4 버전을 삭제(npm uninstall ...)하고, 안정적인 v3 버전을 명시하여 재설치함 (npm install \-D tailwindcss@3 ...).  
  * 설치 정보가 꼬였을 때는 node\_modules 폴더와 package-lock.json 파일을 과감히 삭제하고 npm install로 재설정하는 방법(Clean Install)을 익힘.

### **2\. TypeScript 모듈 인식 및 문법 오류**

* **애로사항:**  
  * src/types.ts 파일을 생성하고 데이터를 정의했으나, NewsCard.tsx 컴포넌트에서 ../types 모듈을 찾을 수 없다는 에러 발생.  
  * 파일 경로는 정확했으나 IDE(VS Code)가 파일을 인식하지 못하거나 문법 규칙이 엄격하게 적용됨.  
* **해결 방안:**  
  * VS Code의 **TypeScript 서버를 재시작**하여 IDE 캐시 문제를 해결.  
  * 최신 Vite/TS 환경의 권장 사항에 따라, 단순히 값을 가져오는 것이 아니라 타입(Interface)을 가져올 때는 **import type { ... }** 문법을 사용하여 명시적으로 선언함.

### **3\. 컴포넌트 간 이벤트 연결 (Props 전달)**

* **애로사항:**  
  * 헤더(Header) 컴포넌트에 있는 '설정' 버튼을 눌러도 설정 모달창이 열리지 않음.  
  * 초기에는 App.tsx에서 \<Header /\> 컴포넌트를 \<div\> 태그로 감싸고 그 \<div\>에 onClick 이벤트를 부여했으나, 헤더의 스타일(Sticky/Fixed 등)로 인해 클릭 이벤트가 제대로 전달되지 않음.  
* **해결 방안:**  
  * React의 정석적인 데이터 흐름인 **Props** 방식을 적용.  
  * Header 컴포넌트 내부에 onOpenSettings라는 함수형 Prop을 뚫어주고, 부모(App)가 자식(Header)에게 직접 실행할 함수를 전달하도록 구조를 개선함.

### **4\. 프론트엔드 데이터 관리 로직 구현**

* **구현 과제:**  
  * 단순히 화면만 보여주는 것이 아니라, '데이터 유지(새로고침 시 저장)'와 **'실시간 업데이트(자동 갱신)'** 기능이 필요했음.  
* **해결 방안:**  
  * **로컬 스토리지(LocalStorage):** useEffect를 활용해 앱 실행 시 저장된 데이터를 불러오고, 데이터 변경 시마다 자동으로 저장하는 로직 구현.  
  * **폴링(Polling):** setInterval을 사용하여 일정 시간(5초)마다 데이터를 요청하고, prev 상태를 활용해 기존 리스트의 앞부분에 새 데이터를 추가하는 로직 구현.

### **5\. 백엔드 API 구조 변경 및 데이터 매핑 (Data Mapping)**

**애로사항:**

* 기존 Node.js 백엔드(포트 4000)에서 Python FastAPI(포트 8000, 배포 서버)로 교체하는 과정에서 뉴스 리스트가 화면에 출력되지 않고 에러가 발생함.  
* 백엔드 개발자(또는 기획자)가 제공한 데이터 구조가 프론트엔드에서 정의한 NewsItem 타입과 일치하지 않음 (예: originalUrl 대신 url 사용, id 및 press 필드 누락 등).

**원인 분석:**

* 프론트엔드는 완성된 형태의 객체를 기대했으나, 백엔드는 최소한의 데이터만 제공하거나 응답 구조를 객체({ news: \[...\] })로 감싸서 보내는 등 **API 명세(Interface)의 불일치**가 원인.  
* TypeScript 환경에서 any 타입을 사용할 수 없도록 설정되어 있어, 유연한 대처가 어려웠음.

**해결 방안:**

* **Adapter 패턴 적용:** API 응답 데이터를 그대로 사용하지 않고, 프론트엔드에서 필요한 형태로 변환해주는 **매핑(Mapping) 로직**을 fetchNewsFromApi 함수에 구현함.  
* **Fallback 처리:** item.press || "News API"와 같이 백엔드에 없는 데이터는 기본값(Default Value)을 설정하여 UI가 깨지지 않도록 방어 코드 작성.  
* **인터페이스 분리:** 백엔드에서 오는 날것의 데이터 타입을 RawNewsItem으로 정의하고, 이를 UI용 NewsItem으로 변환하는 과정을 명시하여 타입 안정성 확보.

### **6\. 오토 폴링(Auto-Polling) 시 데이터 중복 적재**

**애로사항:**

* 5초마다 최신 뉴스를 가져오는 오토 폴링 기능 활성화 시, 화면에 똑같은 뉴스가 계속해서 쌓이는 현상 발생.  
* 분명히 중복 제거 로직(filter)이 있었음에도 불구하고 걸러지지 않음.

**원인 분석:**

* 백엔드에서 고유 id를 주지 않아 프론트엔드에서 Date.now()를 사용해 임의로 ID를 생성함.  
* 매 폴링 시점마다 시간이 달라지므로, **내용(제목)은 같아도 ID가 다른 새로운 뉴스**로 인식되어 리스트에 추가되는 논리적 오류 발생.

**해결 방안:**

* **중복 비교 기준 변경:** 고유하지 않은 id 대신, 기사의 내용적 고유성을 대변하는 title(제목)을 기준으로 중복 검사를 수행하도록 로직 수정.  
* \!prev.some(p \=\> p.title \=== n.title) 조건을 사용하여 이미 존재하는 제목의 기사는 리스트 추가 대상에서 제외함.

### **7\. 다크 모드 시인성 및 스타일 충돌**

**애로사항:**

* 다크 모드를 적용했으나, input 입력창, 설정 모달, 토스트 메시지 등 일부 UI 요소의 글자가 배경색과 비슷하여 보이지 않거나(흰 배경에 흰 글씨 등), 스타일이 어색하게 적용됨.  
* OS 시스템 설정과 사용자의 수동 설정(토글 버튼)이 충돌하여 다크 모드가 해제되지 않거나 고정되는 현상 발생.

**원인 분석:**

* Tailwind CSS의 darkMode 설정이 기본적으로 시스템 설정을 따르도록 되어 있었음.  
* input이나 modal 같은 컴포넌트에 dark: 클래스(예: dark:bg-gray-800, dark:text-white)를 명시적으로 지정하지 않아 브라우저 기본 스타일이 적용됨.

**해결 방안:**

* **Class 전략 사용:** tailwind.config.js에 darkMode: 'class' 옵션을 추가하여, 사용자가 버튼을 눌렀을 때만 HTML 태그에 dark 클래스가 붙도록 제어권을 가져옴.  
* **상세 스타일링:** 검색창(input), 모달(SettingsModal) 등 모든 UI 컴포넌트에 다크 모드용 배경색, 글자색, 테두리 색상 코드를 전수 조사하여 추가함.

### **8\. 프론트엔드-백엔드 기능 구현 속도 차이 (Dependency Issue)**

**애로사항:**

* 프론트엔드에서는 '요약 길이 조절', '기사 수동 추가(POST)' 기능을 UI로 구현했으나, 백엔드 API가 아직 해당 기능을 지원하지 않아 404 에러나 500 에러가 발생.  
* 발표 및 시연이 임박한 상황에서 백엔드 개발을 기다릴 수 없는 상황 발생.

**원인 분석:**

* 기획 단계의 기능 명세와 실제 백엔드 개발 진척도 간의 차이(Dependency) 발생.

**해결 방안:**

* **Mocking & Stubbing:** '기사 수동 추가' 기능은 실제 API 호출이 실패할 경우, 프론트엔드 리스트에만 임시로 추가하여 기능이 작동하는 것처럼 보이게 하는 **하이브리드 처리(Try-Catch)** 적용.  
* **Feature Flagging (UI 비활성화):** 구현되지 않은 '요약 길이 조절' 기능은 과감히 UI에서 제거하거나, 사용자에게 "추후 제공될 기능입니다"라는 토스트 메시지를 띄워 UX 흐름을 끊지 않으면서 개발 상황을 알리는 방식으로 대처.
